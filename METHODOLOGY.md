## Инструкции для Claude Code

Ты — старший инженер-архитектор. Этот файл определяет как ты работаешь в данном проекте.

**Обязательные правила поведения:**
- Перед написанием кода убедись, что для модуля есть спецификация. Нет спецификации — запроси её или создай по шаблону из этого файла
- При старте каждой сессии: прочти `PROGRESS.md` (если существует), чтобы понять текущее состояние проекта
- Не задавай уточняющих вопросов, если ответ есть в документах проекта — найди сам
- При изменении требований — сначала обнови `SPEC.md`, потом пиши код
- Запускай `/security-review` перед каждым значимым коммитом
- Используй Context7 MCP для проверки актуальности документации библиотек перед написанием кода

## Философия: Spec-First

**Ни одна строка кода не пишется, пока модуль не описан в спецификации.**

Спецификация — это ТЗ для агентов, а не для людей. Чем конкретнее описаны таблицы, эндпоинты, компоненты — тем автономнее и точнее работает Claude Code.

| Без спецификации | Со спецификацией |
|-----------------|-----------------|
| Вопросы на каждом шаге | Автономная работа |
| Конфликты между модулями | Модули стыкуются по архитектуре |
| Переписывание через 3 дня | Итерации — улучшение, не переделка |
| Ручной контроль каждой строки | Субагенты работают параллельно |

## Слой 1 — Документ идеи (`PROJECT_IDEA.md`)

Пишется для AI. Обязательные разделы:

| Раздел | Требование |
|--------|-----------|
| **Проблема** | Конкретная боль с числами: «тратит 3 часа вместо 5 минут» |
| **Решение** | Пошаговый процесс — каждый шаг станет модулем |
| **Почему сейчас** | Тайминг рынка, технологии, ниша |
| **Аудитория** | Основная и вторичная, их текущий инструментарий |
| **Архитектура** | Слои системы с диаграммой, стек с обоснованием |
| **Монетизация** | Планы, цены, гейтинг фич (AI генерирует таблицы подписок) |
| **Конкуренты** | Таблица: конкурент → что делает → чего не хватает |
| **План запуска** | Фазы с метриками, MVP выделен отдельно от v2 |
| **Риски** | Таблица: риск → вероятность → митигация |
| **Техдетали** | Структура репо, ключевые таблицы, AI-пайплайн (если есть) |

> ⚠️ **Правило конкретики**: вместо «продвинутая аналитика» → «дашборд с retention по когортам, конверсия free→pro, WAU/MAU из Supabase через Edge Function cron». Никаких абстракций — AI с ними не работает.

## Слой 2 — Техническая спецификация (`SPEC.md`)

Каждый модуль — 6 обязательных блоков. Без них модуль не считается описанным.

### Блок 1 — User Stories
```
Как [роль], я хочу [действие], чтобы [результат]
```
Минимум 3–5 историй: основной сценарий + крайние случаи.

### Блок 2 — Модель данных (формат SQL)
```sql
users: id (uuid PK), email (text UNIQUE NOT NULL), plan (text DEFAULT 'free'), created_at (timestamptz)
RLS: SELECT: auth.uid() = id | UPDATE: auth.uid() = id
```

### Блок 3 — API / Server Actions
```
POST /auth/signup  {email, password}  → 201 | 400 | 409
POST /auth/login   {email, password}  → 200 + session | 401
GET  /api/user     —                  → 200 + {user} | 401
```

### Блок 4 — Экраны и компоненты
Страница → компоненты → действия → все состояния: `loading | error | empty | success`

### Блок 5 — Бизнес-логика
Правила валидации, ограничения, формулы, условия переходов между состояниями.

### Блок 6 — Крайние случаи
Нет данных? API не отвечает? Конкурентное редактирование? Дублирующий webhook? Каждый сценарий — с ожидаемым поведением.

> ⚠️ **Правило формата**: типы полей — как в SQL (`uuid`, `text`, `jsonb`, `timestamptz`). Эндпоинты — как в коде (`POST /api/projects`). Чем ближе к коду — тем точнее генерация.

### Правило обновления спецификации
Если требования изменились в процессе разработки:
1. Сначала обновить `SPEC.md` — зафиксировать изменение
2. Обновить `PROGRESS.md` — отметить что меняется и почему
3. Только потом вносить изменения в код
4. Никогда не переписывать код «по памяти» без обновления спеки

## Слой 3 — Конфигурация (`CLAUDE.md`)

### Ограничение: ≤ 120 строк
Claude Code перечитывает `CLAUDE.md` **при каждом запуске** и **рекурсивно** — из корня и всех поддиректорий. Это значит:
- `CLAUDE.md` в корне + `src/CLAUDE.md` = оба в контексте одновременно
- Раздутые файлы засоряют контекст и снижают качество ответов
- Детали → в `.claude/rules/*.md` и `.claude/skills/*.md`

### Структура `CLAUDE.md`
```
1. Обзор проекта (3–5 строк)
2. Стек технологий
3. Архитектура (кратко)
4. Правила (ссылки на .claude/rules/)
5. Команды запуска / деплоя
```

### Структура репозитория

**Универсальная (стек-нейтральная):**
```
project-root/
├── CLAUDE.md              # конфиг агента (≤120 строк)
├── METHODOLOGY.md         # этот файл
├── PROJECT_IDEA.md        # документ идеи
├── SPEC.md                # техническая спецификация
├── PROGRESS.md            # живое состояние проекта
├── .claude/
│   ├── agents/            # субагенты
│   ├── rules/             # контекстные правила (glob-паттерны)
│   └── skills/            # навыки для повторяющихся задач
├── src/                   # код приложения
└── .github/
    └── workflows/
        └── security-review.yml
```

**Дополнения для Supabase-проектов:**
```
├── supabase/
│   ├── migrations/        # SQL-миграции
│   └── functions/         # Edge Functions
```

**Дополнения для Python-бэкенда:**
```
├── app/                   # FastAPI / Django / Flask
├── alembic/               # миграции
└── requirements.txt
```

| `database-architect` | `opus` | Схема БД, миграции, RLS, индексы | Read, Write, Edit, Bash |

## Слой 5 — PROGRESS.md (управление контекстом между сессиями)

**Самый важный файл для длинных проектов.** Claude Code не помнит прошлые сессии — `PROGRESS.md` — это его внешняя память.

### Структура `PROGRESS.md`
```markdown
# Состояние проекта

## Последнее обновление
[дата и краткое описание последней сессии]

## Что сделано
- [x] Модуль аутентификации — завершён
- [x] БД: таблицы users, projects — созданы и задеплоены
- [ ] API: эндпоинты проектов — в работе

## Сейчас в работе
[Конкретная задача, на которой остановились]

## Следующий шаг
[Что делать в следующей сессии — чётко и конкретно]

## Известные проблемы
[Баги, технический долг, спорные решения]

## Архитектурные решения
[Важные решения принятые в процессе и их обоснование]
```

> ⚠️ **Правило**: в конце каждой сессии обновить `PROGRESS.md`. В начале следующей — прочитать его первым делом.

## MCP-серверы

Каждый субагент обязан проверять актуальность API через Context7 перед написанием кода.

### Обязательные для большинства проектов

| MCP | Назначение | Команда подключения |
|-----|-----------|-------------------|
| **Context7** | Актуальная документация библиотек | `claude mcp add context7 -- npx -y @upstash/context7-mcp` |
| **Supabase** | Прямая работа с БД, миграции | `claude mcp add supabase -- npx -y @supabase/mcp-server-supabase --project-ref <ref>` |
| **GitHub** | Репозиторий, PR, issues | `claude mcp add github -- npx -y @github/mcp-server` |

### Опциональные

| MCP | Назначение |
|-----|-----------|
| **Playwright** | E2E тестирование браузера |
| **Filesystem** | Работа с локальными файлами вне проекта |
| **Stripe** | Прямое управление платёжными объектами |

> ⚠️ **Без Context7** агент пишет код по устаревшим API из тренировочных данных. Подключать обязательно.

## Безопасность: Security-First

Безопасность — не финальный этап, а часть каждого модуля.

### Расписание проверок

| Момент | Действие |
|--------|----------|
| После каждого модуля | Целевой промпт по соответствующему аспекту |
| Перед коммитом | `/security-review` в Claude Code |
| На каждом PR | GitHub Action автоматически |
| Перед деплоем | Полный аудит + чеклист ниже |

### 7 аспектов безопасности

**1. Аутентификация**
- Пароли: bcrypt/argon2 — никогда md5/sha1
- Сессии: TTL + инвалидация при логауте
- Rate limiting на login: ≤5 попыток/мин на IP
- Email-верификация для новых аккаунтов
- Токены сброса пароля: одноразовые, истекают через 1 час
- Auth-секреты: никогда в клиентском коде
- OAuth-редиректы: валидируются (нет open redirect)
- Middleware: проверяет сессию на каждом защищённом маршруте

**2. Авторизация и IDOR**
- RLS включена на каждой таблице: `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`
- RLS покрывает все операции: SELECT, INSERT, UPDATE, DELETE
- Ownership check в каждом API-эндпоинте
- `auth.getUser()` — только на сервере, не доверяем клиенту
- SELECT-запросы: `WHERE user_id = auth.uid()`
- Server Actions и Route Handlers: проверка auth перед каждой операцией

**3. Валидация ввода**
- Zod-схема на каждом серверном эндпоинте
- SQL: только параметризованные запросы, никакой конкатенации строк
- HTML-вывод: экранирование пользовательского контента
- Загрузки: проверка MIME-типа, размера, расширения
- Числовые параметры (limit, offset): верхняя граница обязательна
- UUID вместо последовательных числовых ID

**4. Секреты и API-ключи**
- `.env` в `.gitignore` — всегда
- `.env.example` — только плейсхолдеры, никаких реальных значений
- `SUPABASE_SERVICE_ROLE_KEY` — только серверный код (обходит ВСЕ RLS)
- `NEXT_PUBLIC_*` — никаких секретов (всё видно в браузере)
- `ANTHROPIC_API_KEY` — только на сервере
- Токены: httpOnly cookies, не localStorage
- Git-история: проверить на секреты (`git log -p | grep -i key`)

**5. Rate Limiting**
- Auth: ≤5 попыток/мин на IP
- Регистрация: ≤3 аккаунта/час на IP
- AI-генерация: лимиты по плану подписки (free: 5/день, pro: 50/день)
- API: ≤100 запросов/мин на пользователя
- Вебхуки: проверка подписи (SHA-256 HMAC)
- Body size limit: 1MB для API, 10MB для загрузок
- Реализация: Upstash Redis + `@upstash/ratelimit` для serverless

**6. Деплой и инфраструктура**
- HTTPS принудительный + HSTS-заголовок
- Секреты: Vercel Environment Variables / Supabase Vault
- БД: прямой доступ из публичного интернета заблокирован
- CORS: только ваш домен, никакого wildcard `*`
- Security headers: `X-Frame-Options`, `X-Content-Type-Options`, `CSP`
- Логирование: auth-события, API-ошибки 4xx/5xx, подозрительная активность
- Бэкапы БД: включены и проверены
- Preview deployments: изолированы от продакшн БД

**7. Зависимости**
- `npm audit` перед каждым деплоем
- Нет известных CVE в пакетах
- Context7 MCP: проверка актуальности API перед использованием

### GitHub Action — автопроверка каждого PR

```yaml
# .github/workflows/security-review.yml
name: Security Review
permissions:
  pull-requests: write
  contents: read
on:
  pull_request:
jobs:
  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.event.pull_request.head.sha }}
          fetch-depth: 2
      - uses: anthropics/claude-code-security-review@main
        with:
          comment-pr: true
          claude-api-key: ${{ secrets.CLAUDE_API_KEY }}
```

> ⚠️ Включи «Require approval for all external contributors» в настройках репо — Action не защищён от prompt injection во входящих PR.

## Антипаттерны

| ✗ Антипаттерн | Последствие |
|---------------|-------------|
| Код без спецификации | Фрагментарный код, полная переделка через день |
| Абстрактная спецификация | AI не знает что генерировать |
| `CLAUDE.md` более 120 строк | Засорение контекста при каждом запуске |
| Нет `PROGRESS.md` | Агент теряет контекст между сессиями, дублирует работу |
| Изменение кода без обновления `SPEC.md` | Спека расходится с кодом, новые модули конфликтуют |
| Все субагенты на Opus | Лишние расходы без выигрыша в качестве |
| `qa-reviewer` с правами Write/Edit | Начинает «исправлять» вместо описания проблем |
| Плоский дамп файлов без структуры | Субагенты не находят нужное |
| Без Context7 MCP | Код по устаревшим API из тренировочных данных |
| `SUPABASE_SERVICE_ROLE_KEY` на клиенте | Любой пользователь получает полный доступ к БД |
| Числовые ID вместо UUID | IDOR-атаки через перебор ID |
| Конкатенация строк в SQL | SQL-инъекции |
| Секреты в `NEXT_PUBLIC_*` переменных | Видны всем в браузере через DevTools |
| Нет rate limiting на AI-эндпоинтах | Один бот исчерпывает бюджет API за минуты |
